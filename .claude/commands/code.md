# /code — Implementation from Verified Specification

Generate production code for all components: Dafny compiler output for Prove-tier
components, PRD-backed LLM generation for Direct-tier components.

## Prerequisites

- `docs/PRD.md` must exist and be approved, including a Target Language field.
- For every component marked **Prove** in Section 9: `logic/<component>.dfy` must exist
  and have passed `dafny verify` with zero errors. If any Prove component lacks a
  verified spec, stop and redirect to `/prove`.
- Components marked **Direct** have no Dafny prerequisite.

## Steps

1. **Read `docs/PRD.md` Section 9** (Verification Scope). List all components and their
   tier (Prove or Direct). Read the Target Language field from the PRD header.

2. **Generate code for all components.**
   - **Single component:** generate directly.
   - **Multiple components:** launch one subagent per component in parallel. Each subagent
     receives the component name, its tier, the target language, and the relevant source
     (`.dfy` file for Prove, PRD sections for Direct). Wait for all subagents to finish
     before step 3.

   For each **Prove component** (whether direct or subagent):
   - Compile the verified Dafny spec using the Dafny compiler:
     ```bash
     dafny translate <target> logic/<component>.dfy --output src/
     ```
     where `<target>` is the Dafny language code for the PRD's Target Language:

     | Target Language       | Dafny target code |
     |-----------------------|-------------------|
     | Python                | `py`              |
     | Go                    | `go`              |
     | Java                  | `java`            |
     | JavaScript/TypeScript | `js`              |
     | C#                    | `cs`              |
     | Rust (experimental)   | `rs`              |

   - Do not manually edit the compiler output. It is isomorphic to the spec by
     construction. If the output needs project-specific adaptation (e.g. naming
     conventions, file layout), make only structural changes — no business logic.
   - Add a header comment to the generated file(s):
     ```
     // Generated by Dafny compiler from logic/<component>.dfy — do not edit manually.
     ```
   - Annotate the traceability entry: `dafny:logic/<component>.dfy`

   For each **Direct component** (whether direct or subagent):
   - Read the relevant PRD sections (operations, invariants, edge cases).
   - Write production code to `src/<component>.*`.
   - Map every PRD operation to a function; every failure mode to an explicit return type.
   - No business logic not described in the PRD may be introduced.
   - Annotate each function: `# Implements: docs/PRD.md §<section>`

3. **Write `docs/TRACE.md`** after all components are generated:

   | `src/` function | Source | PRD requirement |
   |---|---|---|
   | `payment.transfer()` | `dafny:logic/payment.dfy` | PRD §4.1 |
   | `dashboard.render()` | PRD direct | PRD §4.3 |
   | `auth.checkPermission()` | `dafny:logic/auth.dfy` | PRD §4.2 |

   Every function in `src/` must appear in this table.

4. **Auto-merge is OFF.** Do not commit, merge, or apply changes automatically.
   Present the full diff and wait for explicit human approval.

5. **Summarise** — list what was generated, distinguish Dafny-compiled (Prove-backed)
   from LLM-generated (Direct-backed) functions, and flag any manual review items.

## Language-Specific Notes (Direct-tier only)

The following notes apply to Direct-tier components, where code is LLM-generated.
Prove-tier components are compiled by Dafny and do not require these guidelines.

### Python (Tier 1)
- Use type hints throughout.
- Express invariants as `assert` statements guarded by `__debug__`.
- Use `dataclasses` or `NamedTuple` for structured types.
- Compatible with numpy/pandas for data science workloads.

### Rust (Tier 1)
- Use `Result<T, E>` for operations with error cases.
- Map invariants to `debug_assert!` macros.
- Use enums for sum types.
- Apply `#![forbid(unsafe_code)]` unless unsafe is required and proved in the spec.

### TypeScript/JavaScript (Tier 1)
- Use TypeScript types to encode type constraints.
- Map preconditions to runtime guards with typed errors.

### Go (Tier 1)
- Return `(T, error)` pairs for operations with error cases.
- Use struct methods to group related operations.

### Java, C, C++ (Tier 1)
- Standard idiomatic patterns apply.
- Use assertions for debug-mode invariant checking.

### Tier 2 languages
- Follow the same mapping principles; raise any idiomatic gaps as review items.

### Tier 3 languages
- Direct structural translation; minimal idiomatic adaptation.
- Flag all non-obvious constructs for manual review.

## Constraints
- For Prove components: use the Dafny compiler. Do not generate Prove-tier code by hand.
- For Direct components: no business logic not described in the PRD may be introduced.
- The traceability summary is mandatory, not optional.
- If the PRD target language is not a Dafny compile target and Prove-tier components exist,
  stop and return to `/define` to resolve the conflict.
